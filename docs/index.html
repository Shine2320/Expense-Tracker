Expense Tracker

A local-first Flutter expense tracker with credit-card repayment accounting and bill splitting. Material 3, Riverpod, Hive. No network, no accounts, no telemetry.

Overview

Expense Tracker records what you spend against a monthly salary and tells you what's left. Two things make it more than a list of numbers, and both are where its complexity lives:

  • Credit-card expenses are counted in the month you pay the card, not the month you swiped it. An unpaid card expense counts against no month at all.
  • Split expenses let you front a bill for other people. Only your share counts against your salary, and when someone repays you, the month you originally paid gets corrected.

Everything is stored on-device in Hive. There is no INTERNET permission in the Android manifest, no analytics, and no API keys anywhere in the tree.

Navigation

There is no router and no named-route table. Top-level tabs are an IndexedStack in HomeShell; everything else is an imperative Navigator.push(MaterialPageRoute(...)) or a showModalBottomSheet. Adding a screen means pushing it from wherever you need it.

Theme

AppTheme builds light and dark from one seed colour, and installs MoneyColors as a ThemeExtension. Widgets read Theme.of(context).textTheme, colorScheme, and context.money — never raw hex, never a bare fontSize:.

// Semantic, and correct in both light and dark:
Text(amount, style: MoneyText.medium(context)
    .copyWith(color: context.money.forBalance(remaining)));

MoneyColors carries four semantic roles — positive (money kept), negative (money spent), pending (owed, e.g. an unpaid card), settled — each with a container/on-container pair for chips.

Business rules

Read this section before changing any number on any screen Every serious bug this app has had came from a screen inventing its own arithmetic instead of using the rules below. If you are summing expense.amount in a widget, you are almost certainly wrong.

1. The accounting date — which month an expense belongs to

expense_repository.dart → getAccountingDate()

DateTime? getAccountingDate(ExpenseModel expense) {
  if (expense.isDeleted) return null;          // counts nowhere
  if (!expense.isCreditCard) return expense.date;
  if (!expense.isPaid) return null;            // unpaid card: counts NOWHERE
  return expense.repaymentDate ?? expense.date; // the month you PAID the card
}
ExpenseCounts inWhy
Cash / debitIts own dateThe money left when you spent it
Credit card, unpaidNowhereYour salary hasn't been touched yet — it's a liability, shown as "Card due"
Credit card, paidThe repayment monthThat's the month your salary actually paid for it
Deleted (struck out)NowhereSoft-deleted rows stay in the box but count for nothing

So a card expense from 20 May that you pay on 3 June counts against June's salary. It appears in June's summary labelled "Credit paid this month · Original expense: 20 May".

2. The counted amount — how much an expense actually costs you

expense_repository.dart → getNetSplitAmount() / getCountedAmount()

For a split you fronted, only your share is yours. Every participant who has repaid you is subtracted:

net = expense.amount − Σ(paid, non-slip-payer participants)   // floored at 0
MethodReturnsUse for
getNetSplitAmount(e)Net cost, ignoring whether it's counted yet"What do I still owe on this card?" — pending card totals
getCountedAmount(e)Net cost, or 0 if it counts in no monthAnything that must agree with a month's total
These two are not interchangeable getCountedAmount() returns 0 for a pending card expense (it has no accounting date). Use it for a "card due" figure and the screen will always read zero. Use getNetSplitAmount() there instead.

3. The carry-over chain — how months link together

balance_repository.dart → rebuildCarryOverChain()

Each month's balance is salary + carryOver − totalExpenses, and each month's carry-over is the previous month's remainder:

void rebuildCarryOverChain() {
  // "YYYY-MM" is zero-padded → lexicographic sort == chronological.
  final balances = _box.values.toList()..sort((a, b) => a.id.compareTo(b.id));
  var previousRemaining = 0.0;
  for (final balance in balances) {
    balance.carryOver = previousRemaining + balance.carryOverAdjustment;
    _box.put(balance.id, balance);
    previousRemaining = balance.remainingBalance;  // read AFTER assigning
  }
}

Properties that are load-bearing, not incidental:

  • The chain runs after totals, always. Each month's remainder feeds the next, so totalExpenses must be final before the walk starts.
  • Negatives roll forward. Overspend May and June starts in the hole. The same wire that carries an improvement forward must carry a shortfall, or money is created from nothing.
  • Gap months pass through. Only existing rows are walked; a month with no salary and no expenses is a no-op, so May's remainder correctly reaches July. Rows are never synthesised.
  • The earliest month is its own opening balance. It has no predecessor, so previousRemaining is 0 and its carry-over is exactly its adjustment. No special case needed.

Manual carry-over is an adjustment, not an override

When you type a carry-over, the app stores the difference from the calculated value in carryOverAdjustment, not the number itself:

Calculated from last month:  3000
You type:                    3200   →  carryOverAdjustment = +200

A friend later repays a 500 split from last month:
Calculated becomes:          3500
Displayed carry-over:        3700   (3500 + your 200)

This is why the field exists. A plain override would freeze the month, and later corrections — the exact thing this app is supposed to track — would never reach it. "Use calculated" in the carry-over dialog sets the adjustment back to 0.

4. Reconciliation — the one true recompute

expense_repository.dart → reconcileMonthlyExpenses()

Rebuilds every month's totalExpenses from scratch, then rebuilds the carry-over chain. It runs on app start, after every split mutation, and after every import. It is idempotent — running it twice yields byte-identical balances, and there's a test that asserts exactly that.

  1. Heal any paid card expense that has no repaymentDate (persist the fallback, don't re-guess it on every read)
  2. Index splits and participant repayments once — the naive version rescanned both boxes per expense
  3. Sum net per month by accounting date
  4. Write totalExpenses for every month (zeroing months that no longer have any)
  5. rebuildCarryOverChain()
The incremental path chains too _updateMonthlyExpenses() (add / edit / delete / mark-paid) also calls rebuildCarryOverChain(). Rebuilding only inside reconcile would leave later months stale until the next restart — the same class of bug in a new costume.

Storage & migrations

hive_storage.dart opens all five with a backoff retry (0, 300, 800, 1600 ms) — but only for lock errors, which happen when a previous process still holds the file. Any other error rethrows immediately.

There is no migration framework — defaultValue is the migration

To add a field to an existing model:

@HiveField(4, defaultValue: 0.0)   // next unused index; defaultValue REQUIRED
double carryOverAdjustment;

Then re-run build_runner. The generator emits a null-safe read for that field:

carryOverAdjustment: fields[4] == null ? 0.0 : fields[4] as double,

Records written before the field existed decode as the default instead of throwing on a non-null cast. Omit defaultValue and every existing install crashes on first read.

The v2 balance migration

balance_migration.dart, guarded by the balances_migrated_v2 SharedPreferences flag.

Before the carry-over chain existed, carryOver was a snapshot written once when a month was first opened. The chain now derives it — which means the earliest month, having no predecessor, would rebuild to just its adjustment (0) and silently lose its opening balance. So on first launch after upgrading, the migration:

  1. Exports a JSON backup to Downloads (a failure here is reported, not fatal — see below)
  2. Seeds the earliest month's carryOverAdjustment from its stored carryOver
  3. Sets the flag, and shows "Balances recalculated" on the dashboard
Ordering is not negotiable The migration must run before the first reconcileMonthlyExpenses() of the process. Once the chain has zeroed the earliest month's carryOver, there is nothing left to seed from and the opening balance is gone for good. That's why a failed backup does not abort the seed.

Only the earliest month is seeded, deliberately. Seeding later months would preserve their frozen (wrong) values as permanent adjustments and defeat the entire fix.

Export & import

data_export_import.dart

  • JSON — map-driven, so it picks up new model fields automatically once toMap/fromMap know about them.
  • Excel — read positionally. New columns must be appended; inserting one silently corrupts every field after it. Readers guard on row.length so older backups still import.
  • Both paths finish with a full reconcile.
  • Android saves via a MethodChannel to MediaStore; elsewhere it falls back to path_provider.
Import is destructive, not additive _deleteMissingKeys() removes any record not present in the imported file. Importing a backup makes your data match that file — it does not merge.

Why the expense list is intentionally different Every other money surface uses accounting dates and counted amounts. The expense list does not — on purpose. It isn't month-scoped (it queries all time), so it can't contradict the balance card; and switching it to accounting semantics would make every pending card expense vanish from the list, deleting the very rows you go there to manage. It's a transaction ledger: transaction amount, at transaction date. Its day totals say "Spent", not "Total", for the same reason.

Testing

flutter test — 24 tests across two files. Each seeds a temp Hive directory, registers adapters, opens boxes, and tears down.

  • test/expense_credit_card_test.dart — 22 accounting tests, run on the Dart VM.
  • test/dashboard_settlement_flow_test.dart — 2 widget tests driving the dashboard end-to-end.
Two rules for widget tests here, both learned the hard way
  1. Wrap every Hive call in tester.runAsync(). testWidgets runs its body in a fake-async zone; Hive does real file I/O, so those futures never complete and the test hangs until timeout with a bare "did not complete".
  2. Don't use pumpAndSettle() on the dashboard — it never reaches a quiescent frame, so settling spins out. Use bounded pump(Duration) calls.

What's covered

  • Credit repayment month — single and bulk card payment land in the repayment month; undo removes them from it; card-name normalisation; reconcile re-homes existing paid expenses.
  • Carry-over chain — propagation, negative roll-forward, gap months, earliest-month opening balance, idempotence, and the incremental path chaining without a reconcile.
  • Splits — settling reduces the original month; undo restores it; settling a previous month's split raises this month's remaining (the headline regression test); a manual adjustment survives a later settlement; edit round-trips preserve participant id and paidAt.
  • Soft delete — striking out an expense updates its month.
Three tests assert on source text, not behaviour The first three read screen .dart files off disk and assert on substrings (e.g. expect(summarySource, isNot(contains('+ expense.amount')))). They break on a reformat and pass on a rename, they depend on the CWD being the repo root, and they give false confidence that the UI is covered. Replace them with behavioural tests as those screens change; don't extend the pattern.
The headline regression test dashboard remaining rises when a previous month split settles reproduces the original bug through the real widget tree: front a split last month, settle it this month, and this month's remaining must rise with no restart. It fails on the pre-fix code.

Known issues

Blocks release

  • Release builds are signed with the debug key. build.gradle still has signingConfig signingConfigs.debug in the release block. Play will reject it. Checklist Step 4 has the fix.
  • Package is still com.example.expense_tracker — com.example.* cannot be published, and the id is permanent once live. Same for the iOS bundle id com.example.expenseTracker.

Platform limits

  • The web target cannot compile despite web/ existing — data_export_import.dart imports dart:io.
  • Desktop export is untested. The MediaStore MethodChannel is implemented only in MainActivity.kt; other platforms fall back to path_provider.
  • The dashboard never settles under the test binding — something on it schedules frames indefinitely. It doesn't affect the app, but it means pumpAndSettle() can't be used there. Worth tracking down.
Fixed: the win32 build break The stale sdk: '>=3.2.3' floor forced pub to resolve win32 5.2.0, which calls UnmodifiableUint8ListView — removed in Dart 3.4. It broke flutter build windows and every test that imported a screen. Resolved by raising the SDK floor to 3.4.0 and upgrading file_picker 6 → 11 (which made FilePicker static: FilePicker.pickFiles(), no more .platform).

Design limits

  • A "month" is always a calendar month, in device-local time. There is no payday or cycle-start setting. If you're paid on the 25th, the app can't model that.
  • setMonthSalary(monthKey, ...) exists but nothing calls it — salary is only editable for the current month, so backfilling an old month's salary isn't reachable from the UI.
  • Split amounts aren't validated on import. The add-expense form checks participants sum to the total (±0.01), but a hand-edited backup can bypass it. Netting floors at 0, so an overpayment silently vanishes rather than becoming a credit.
  • paidAt doesn't move the accounting month. A split settled in June reduces May — deliberately, so past months stay truthful — with the UI labelling it. Its credit-card sibling (repaymentDate) works the opposite way. That asymmetry is intentional but surprising.
  • No localization. All UI text is hardcoded English; intl is used only for number/date formatting. Currency locales affect formatting, not language.
